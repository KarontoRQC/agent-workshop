# Agent Workshop

智能体工作坊网页原型，采用 `React + Vite` 前端和 `Flask` 后端骨架。

当前主体验证方向是「智能体知识图谱 / 智能体星图」：用强视觉、强演示感的空间图谱，把用户的业务问题逐步问诊成一组可直接使用的智能体组合，并服务于智能体工作坊现场演示和后续交付包装。

## 需求来源

本文档根据《智能体工作坊方案及课程设计决策会 - Transcribe.docx》逐字稿整理。逐字稿中包含课程设计、销售表达和网页开发需求；本 README 只沉淀和网页原型直接相关的开发需求，课程交付细节作为背景和后续范围记录。

## 项目定位

这个网页不是普通的智能体列表页，而是一个「聚合平台生成器」的演示原型：

- 用户输入行业、业务问题、当前卡点或目标结果。
- 页面以类似宇宙星图 / 知识图谱 / 抽卡探索的方式，逐步展开关键词、场景、需求和能力节点。
- 前端交互最终沉淀为一组参数或关键词。
- 后端或 Agent 根据这些参数，从已有的 66 个智能体中匹配 6-9 个适合当前问题的智能体。
- 结果以「智能体组合包 / 作战武器库 / 聚合平台」形式呈现，用户可以逐个点击跳转到对应智能体。

现场演示的优先级高于完整生产系统能力：第一目标是足够炫酷、系统感强、能在课程现场完成「问诊出方案」的导演式展示。

## 核心用户体验

### 1. 星图问诊

用户从一个模糊问题开始，例如「我是做白酒行业的，想解决招商获客和卖货问题」。系统需要：

- 识别行业、问题和目标。
- 在星图中生成或高亮一组相关关键词节点，例如行业、渠道、获客、销售、私域、内容、复盘等。
- 允许用户继续点击或补充输入，使星图逐轮聚焦。
- 每次选择都让图谱向前推进一层，形成探索感和路径感。

### 2. 参数沉淀

图谱交互的本质是收集参数，而不是单纯展示动画。前端需要把用户输入和节点选择整理为：

- 行业或业务类型。
- 核心问题。
- 关键需求。
- 目标结果。
- 当前已选图谱路径。
- 可用于智能体匹配的一组关键词。

这些参数用于后续语义匹配和推荐智能体组合。

### 3. 智能体组合推荐

系统从 66 个已有智能体中推荐若干个组合。会议中反复提到的数量是 6-9 个；当前原型可以先展示 3 个或更多，但交互和数据结构应支持扩展到 9 个。

推荐结果需要说明：

- 推荐了哪些智能体。
- 每个智能体解决哪个阶段的问题。
- 为什么推荐它。
- 这些智能体如何共同解决用户提出的业务问题。

### 4. 聚合平台交付

推荐结果不只是列表，而应包装成一个智能体组合平台：

- 每个智能体有名称、阶段、说明、推荐理由和跳转入口。
- 用户可以逐个点击进入对应智能体。
- 后续可支持一键打开多个智能体入口。
- 组合结果要有「这是为你生成的作战武器库」的交付感。

## 功能需求

### 前端

- 提供 Obsidian 风格或宇宙星图风格的知识图谱交互。
- 首屏要有规模感，但不能把所有节点一次性堆满导致不可读。
- 从母节点开始展开第一层行业 / 场景节点。
- 点击节点后，当前节点变成新的焦点，继续展开痛点、任务和智能体能力。
- 路径推进时，需要保留父级路径作为可见锚点，避免用户迷失。
- 左侧或悬浮区域保留 Agent 对话窗口，用于输入业务问题和展示流式回复。
- Agent 回复样式需要呈现为：路径确认、知识图谱路径卡片、推荐智能体组合卡片。
- 右侧推荐栏展示推荐智能体组合，并支持点击跳转。
- 图谱交互与智能体跳转逻辑解耦，避免后续接入真实智能体时重构前端。

### 后端 / Agent

- 接收用户问题、当前焦点节点和已选路径。
- 基于语义匹配从 66 个智能体描述中挑选合适组合。
- 不优先采用硬编码规则；会议中明确倾向使用 Agent 或模糊语义匹配，因为硬编码维护成本更高。
- 支持流式返回：
  - 知识图谱路径确认。
  - 路径解释。
  - 推荐智能体开始生成、字段生成、完成状态。
  - 完整推荐列表。
- 支持把图谱路径解析为前端可播放的节点序列。

### 数据

- 66 个智能体需要有清晰描述，至少包含：
  - 名称。
  - 能力说明。
  - 适用场景。
  - 阶段标签。
  - 跳转链接。
  - 可用于语义匹配的关键词或描述文本。
- 图谱 seed 数据不应包含前端视觉样式字段；坐标、颜色、尺寸等由前端布局层处理。
- 推荐算法可以先使用现有 seed 数据和 agent 描述，后续再接入真实知识库或向量检索。

## 视觉与交互方向

会议中的关键词是「酷」「漂亮」「系统感」「宇宙星图」「知识图谱」「抽卡」「现场问诊出方案」。视觉实现要围绕这些感受服务，但仍保持可读性。

设计约束：

- 风格接近 Obsidian local graph 或空间知识图谱，不要做成线性流程图。
- 图谱应有探索感：输入一次、推进一层、点亮一组关键词。
- 不要一开始展示过多节点。
- 节点关系要有局部焦点，不是全量星空噪音。
- 推荐结果要像「组合包」而不是普通搜索结果。
- 演示时需要让观众感觉系统在实时理解业务、收敛路径、生成方案。

## 版本与跳转策略

会议中提到两个潜在版本：

- 海外 / GPT 版本：智能体入口跳转到 GPTs。
- 国内 / Coze 版本：智能体入口跳转到扣子或国内可访问智能体。

当前演示可以优先完成一个版本；代码层面应保留入口适配能力，使同一套推荐结果可以切换不同平台链接。

## 课程背景与后续范围

逐字稿中还提到两个与网页形态相似、但不属于当前主线的后续方向：

### 流量工作坊内容星图

将原本二维表里的「产品 - 人群 - 需求 - 方案 - 文案」映射成知识图谱。用户从行业、人群或需求进入，最终定位到一篇文案，并可一键跳转到改写类 GPT / Agent 进行「一键爆改」。

这可以作为后续独立模块，不应阻塞当前智能体组合星图。

### 智能体制作 / 调优课程工具

课程中还讨论了带用户做图文内容智能体、咨询类智能体、智能体调优和入企落地 SOP。这些属于课程交付工具链，和当前网页原型有关联但不完全等同。

当前网页只需要为这些课程叙事提供演示支撑：让用户相信「我们有一个能把业务问题分诊成智能体组合的系统」。

## 非目标

- 不要求用户在当前网页内真正创建和部署新智能体。
- 不要求完全还原课程现场的所有教学流程。
- 不要求一开始就实现复杂向量数据库或完整生产级推荐系统。
- 不要求封装 66 个智能体本体；网页只负责推荐、组合和跳转。
- 不把图谱做成静态流程图或普通表格视图。

## 目录

- `frontend/`: React + Vite + PixiJS 交互图谱前端。
- `backend/`: Flask API 骨架，提供系统接口和 Coze 流式接口适配。
- `data/`: 不含视觉样式的图谱 seed 数据和 SQL schema。
- `docs/`: 数据格式、表结构、图谱渲染和 Agent 接入说明。
- `scripts/`: 图谱数据导出脚本。

## 启动前端

```powershell
cd frontend
npm install
npm run dev
```

默认地址：`http://127.0.0.1:5173`。

## 启动后端

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\python -m pip install -r requirements.txt
.\.venv\Scripts\python app.py
```

默认地址：`http://127.0.0.1:5000`。

## 图谱数据

图谱内容数据见 `data/knowledge_graph_seed.json`，它只保留节点、边、智能体和映射关系，不包含坐标、颜色、尺寸等前端样式字段。

详细格式见 `docs/knowledge-graph-database-format.md`。
