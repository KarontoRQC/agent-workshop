# Agent Workshop

智能体工作坊网页原型，采用 `React + Vite` 前端和 `Flask` 后端骨架。

当前主体验证方向是「智能体路径规划 / 智能体星图」：用强视觉、强演示感的空间路径，把用户的业务问题包装成一条实时生成的思考路线，并把推荐结果呈现为一组可直接使用的智能体组合。

## 需求来源

本文档根据《智能体工作坊方案及课程设计决策会 - Transcribe.docx》逐字稿整理。逐字稿中包含课程设计、销售表达和网页开发需求；本 README 只沉淀和网页原型直接相关的开发需求，课程交付细节作为背景和后续范围记录。

## 项目定位

这个网页不是普通的智能体列表页，而是一个「聚合平台生成器」的演示原型：

- 用户输入行业、业务问题、当前卡点或目标结果。
- 路径规划智能体自由生成一条业务理解路线；这条路线不受固定图谱、固定节点或固定边约束。
- 前端只把路线字符串渲染成宇宙星图 / 路径点亮 / 抽卡探索的视觉效果。
- 推荐智能体独立根据用户需求，从 `data/source_agents_full.json` 中的 60 个智能体里推荐适合当前问题的组合。
- 结果以「智能体组合包 / 作战武器库 / 聚合平台」形式呈现，用户可以逐个点击跳转到对应智能体。

现场演示的优先级高于完整生产系统能力：第一目标是足够炫酷、系统感强、能在课程现场完成「问诊出方案」的导演式展示。

## 核心用户体验

### 1. 动态路径规划

用户从一个模糊问题开始，例如「我是做白酒行业的，想解决招商获客和卖货问题」。系统需要：

- 识别行业、问题和目标。
- 由路径规划智能体生成本次对话的路线，例如行业、渠道、获客、销售、私域、内容、复盘等。
- 前端把这条路线临时渲染成点、线和路径聚焦效果。
- 路线只服务于理解和展示，不作为推荐智能体的约束条件。

### 2. 推荐解耦

推荐智能体不从图谱节点上取绑定关系。它只接收：

- 用户原始需求。
- 路径规划阶段生成的路线文本，作为理解参考。
- 60 个可用智能体名称。

然后它根据业务需求独立推荐 1-3 个智能体组合。

### 3. 智能体组合推荐

系统从 60 个已有智能体中推荐若干个组合。当前流式推荐默认展示 1-3 个，但交互和数据结构应支持后续扩展。

推荐结果需要说明：

- 推荐了哪些智能体。
- 每个智能体解决哪个阶段的问题。
- 为什么推荐它。
- 这些智能体如何共同解决用户提出的业务问题。

### 4. 聚合平台交付

推荐结果不只是列表，而应包装成一个智能体组合平台：

- 每个智能体有名称、阶段、说明、推荐理由和跳转入口。
- 用户可以逐个点击进入对应智能体。
- 后续可支持一键打开多个智能体入口。
- 组合结果要有「这是为你生成的作战武器库」的交付感。

## 功能需求

### 前端

- 提供 Obsidian 风格或宇宙星图风格的知识图谱交互。
- 首屏要有规模感，但不能把所有节点一次性堆满导致不可读。
- 不加载固定图谱数据，不从预设母节点展开。
- 路径推进来自路径规划智能体输出的路线字符串。
- 路径动画需要保留视觉上的前后关系，避免用户迷失。
- 左侧或悬浮区域保留 Agent 对话窗口，用于输入业务问题和展示流式回复。
- Agent 回复样式需要呈现为：路径确认、知识图谱路径卡片、推荐智能体组合卡片。
- 右侧推荐栏展示推荐智能体组合，并支持点击跳转。
- 图谱交互与智能体跳转逻辑解耦，避免后续接入真实智能体时重构前端。

### 后端 / Agent

- 接收用户问题和路径规划智能体生成的动态路线。
- 把用户需求、动态路线和 60 个智能体名称传给推荐智能体。
- 不采用固定图谱节点到智能体的硬编码绑定。
- 支持流式返回：
  - 知识图谱路径确认。
  - 路径解释。
  - 推荐智能体开始生成、字段生成、完成状态。
  - 完整推荐列表。
- 支持把动态路线文本解析为前端可播放的临时节点序列。

### 数据

- 60 个智能体需要有清晰描述，至少包含：
  - 名称。
  - 能力说明。
  - 适用场景。
  - 阶段标签。
  - 跳转链接。
  - 可用于语义匹配的关键词或描述文本。
- 项目不再维护预设图谱文件、节点、边或节点到智能体映射。
- 推荐算法使用智能体名称和用户需求，后续可接入真实知识库或向量检索。

## 视觉与交互方向

会议中的关键词是「酷」「漂亮」「系统感」「宇宙星图」「知识图谱」「抽卡」「现场问诊出方案」。视觉实现要围绕这些感受服务，但仍保持可读性。

设计约束：

- 风格接近 Obsidian local graph 或空间知识图谱，不要做成线性流程图。
- 图谱应有探索感：输入一次、推进一层、点亮一组关键词。
- 不要一开始展示过多节点。
- 节点关系要有局部焦点，不是全量星空噪音。
- 推荐结果要像「组合包」而不是普通搜索结果。
- 演示时需要让观众感觉系统在实时理解业务、收敛路径、生成方案。

## 版本与跳转策略

会议中提到两个潜在版本：

- 海外 / GPT 版本：智能体入口跳转到 GPTs。
- 国内 / Coze 版本：智能体入口跳转到扣子或国内可访问智能体。

当前演示可以优先完成一个版本；代码层面应保留入口适配能力，使同一套推荐结果可以切换不同平台链接。

## 课程背景与后续范围

逐字稿中还提到两个与网页形态相似、但不属于当前主线的后续方向：

### 流量工作坊内容星图

将原本二维表里的「产品 - 人群 - 需求 - 方案 - 文案」映射成知识图谱。用户从行业、人群或需求进入，最终定位到一篇文案，并可一键跳转到改写类 GPT / Agent 进行「一键爆改」。

这可以作为后续独立模块，不应阻塞当前智能体组合星图。

### 智能体制作 / 调优课程工具

课程中还讨论了带用户做图文内容智能体、咨询类智能体、智能体调优和入企落地 SOP。这些属于课程交付工具链，和当前网页原型有关联但不完全等同。

当前网页只需要为这些课程叙事提供演示支撑：让用户相信「我们有一个能把业务问题分诊成智能体组合的系统」。

## 非目标

- 不要求用户在当前网页内真正创建和部署新智能体。
- 不要求完全还原课程现场的所有教学流程。
- 不要求一开始就实现复杂向量数据库或完整生产级推荐系统。
- 不要求封装 60 个智能体本体；网页只负责推荐、组合和跳转。
- 不把图谱做成静态流程图或普通表格视图。

## 目录

- `frontend/voice-particle-jarvis/`: React + Vite + Three.js 语音粒子路径前端。
- `backend/`: Flask API 骨架，提供系统接口和 Coze 流式接口适配。
- `data/source_agents_full.json`: 60 个智能体源数据。
- `docs/`: Agent 接入说明。

## 启动前端

```powershell
cd frontend/voice-particle-jarvis
npm install
npm run dev
```

默认地址：`http://127.0.0.1:5173`。

## 启动后端

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\python -m pip install -r requirements.txt
.\.venv\Scripts\python app.py
```

默认地址：`http://127.0.0.1:5000`。

## 智能体数据

智能体候选集见 `data/source_agents_full.json`。路径规划和推荐智能体都不读取固定图谱；前端只把动态路线渲染成视觉效果。
